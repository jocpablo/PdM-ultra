Capas de Seguridad Implementadas
Autenticación
JWT con firma HMAC-SHA256
Tokens con expiración de 8 horas. Sin estado en servidor — revocación por expiración.
Contraseñas con scrypt + salt aleatorio
Función de derivación de clave resistente a GPU. Cada contraseña tiene un salt único de 16 bytes.
Rate limiting en login
Máximo 10 intentos por IP en ventana de 15 minutos. Previene ataques de fuerza bruta.
Comparación de tokens resistente a timing attacks
Se usa crypto.timingSafeEqual() para evitar timing oracles en verificación de contraseñas y JWT.
Control de Acceso (RBAC)
4 niveles de rol: Admin Sistema / Admin Planta / Técnico / Visor
Cada endpoint verifica el rol mínimo requerido antes de ejecutar.
Aislamiento de datos por planta
Cada query filtra automáticamente por las plantas asignadas al usuario. Un visor de Planta A nunca recibe datos de Planta B.
Jerarquía de administración
Admin Planta solo puede crear Técnicos/Visores en sus plantas. Solo el Admin Sistema puede crear plantas y admins.
UI oculta por rol
Botones y secciones con data-min-rol son ocultados automáticamente si el rol no es suficiente.
Auditoría y Trazabilidad
Registro automático de acciones críticas
Login, logout, creación/edición de usuarios, eliminación de equipos y plantas quedan registrados con timestamp e IP.
Audit log visible solo por Admin Sistema
El historial de acciones no es accesible por admins de planta ni técnicos.
Recomendaciones Adicionales (producción)
HTTPS obligatorio
En producción, servir la app detrás de nginx/Caddy con certificado TLS. Sin HTTPS los JWT viajan en claro.
Cambiar JWT_SECRET en .env
Usar un string aleatorio de al menos 64 caracteres. Rotarlo si hay sospecha de compromiso.
Firewall en PostgreSQL
La BD debe ser accesible solo desde localhost o la red interna, nunca desde internet.
Headers de seguridad HTTP
Añadir Helmet.js al servidor para Content-Security-Policy, X-Frame-Options, etc.
